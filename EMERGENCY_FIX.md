# Emergency Fix Flow

紧急修复流程用于生产事故或核心功能不可用场景。它允许先做最小修复，但不允许跳过验证和补记录。

## 1. 适用场景

满足任一条件，可以进入 `emergency_fix_hook`：

- 生产服务不可用。
- 数据错误正在持续扩大。
- 安全风险正在发生。
- 外部 API 变更导致核心路径失败。
- 部署后出现阻塞用户使用的严重回归。

## 2. 不适用场景

以下情况不能使用紧急修复流程：

- 普通功能开发。
- 需求不清导致的返工。
- 想绕过测试或审查。
- 借修 bug 扩大范围。

## 3. 流程

```text
emergency_fix_hook
  -> 记录事故现象
  -> 判定影响范围
  -> ②开发工程师最小修复
  -> 真实验证
  -> 补测试或补验证记录
  -> ③验收工程师确认
  -> 必要时 ⑤安全 / ⑩风险 / ⑧集成
  -> 输出事故修复报告
```

## 4. 必填记录

```markdown
# Emergency Fix Report

## 事故现象
[发生了什么]

## 影响范围
[影响哪些用户、模块、数据]

## 紧急等级
P0 / P1 / P2

## 根因判断
[已确认 / 暂未确认]

## 修复内容
[最小修复说明]

## 验证结果
[真实验证步骤和结果]

## 补充测试
[补了哪些测试；如未补，说明原因和后续时间]

## 风险复核
[安全/风险/集成是否需要复核]

## 后续动作
[复盘、补测试、重构、监控]
```

## 5. 禁止事项

- 禁止无验证宣布恢复。
- 禁止不记录事故影响范围。
- 禁止借紧急修复新增无关功能。
- 禁止 L3 风险修复不补安全或风险复核。
